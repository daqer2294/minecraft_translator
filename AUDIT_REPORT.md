## 1. Архитектура перевода

### 1.1 Где и как вызывается ChatGPT API

Вся интеграция сосредоточена в **`src/translators.py`**. LLM вызывается **напрямую через `urllib.request`** (НЕ через OpenAI SDK), эндпоинт `POST {base_url}/v1/chat/completions`:

- Одиночный запрос: `Translator._request_single` — `src/translators.py:305-337`
- Batch-запрос: `Translator._request_batch` — `src/translators.py:340-378`

Обе функции строят один и тот же URL (`translators.py:306`, `translators.py:341`), кладут `Authorization: Bearer <key>` и `Content-Type: application/json`, шлют POST и читают `data["choices"][0]["message"]["content"]`.

Точки входа в перевод:
- `Translator.translate(text)` — одиночная строка (`translators.py:147-184`)
- `Translator.translate_many(texts)` — список строк с дедупом/чанкингом/батчингом (`translators.py:187-277`)

Процессоры вызывают только эти два метода; они **не знают ни о каком API** — хорошая абстракция на уровне процессоров.

### 1.2 Формат промптов

Промпты захардкожены inline в двух местах:

- **System-prompt одиночного** — `translators.py:308-317`. Просит «professional localization engine», перевести в целевой язык, не трогать плейсхолдеры/namespaced ID/§-коды/JSON, вернуть только текст.
- **System-prompt batch** — `translators.py:343-354`. То же самое, но: «переведи КАЖДЫЙ элемент JSON-массива, сохрани длину и порядок массива, верни только валидный JSON-массив строк».
- User-content батча — `json.dumps(texts, ensure_ascii=False)` (`translators.py:355`).

Модель-параметры: только `model` + `temperature: 0` (`translators.py:318-325`, `356-363`). **`top_p`, `max_tokens`, `response_format`, seed и т.п. не задаются.**

### 1.3 Парсинг и валидация ответа

- Одиночный: просто `.strip()` контента (`translators.py:337`).
- Batch: `_coerce_json_array(raw, expected_len)` — `translators.py:68-105`. Снимает markdown-ограды ` ```json `, пробует `json.loads`, при неудаче вырезает первый `[` … последний `]`. Если длина массива ≠ ожидаемой или парсинг не удался → `RuntimeError("Batch output format mismatch")`.
- **Валидация плейсхолдеров** (пост-проверка): `_extract_tokens` собирает `%s/%1$s/{count}` и namespaced ID из исходника и перевода; если наборы не совпадают и `strict=True`, перевод **откатывается на оригинал** (`translators.py:176-179` для single, `259-264` для batch).

### 1.4 Retry / обработка ошибок / rate limiting

- `_retry_call(func)` — `translators.py:280-302`. До `RETRY_MAX_ATTEMPTS` (6) попыток, экспоненциальный backoff с джиттером (`base 2.0s`, `max 30s`, `jitter 0.25`).
- **Ретраятся только сетевые ошибки и 429** (детект по строке `"429"/"rate limit"/"too many requests"` и типу `URLError/TimeoutError`, `translators.py:290-292`). Логические/парсинговые ошибки — без ретрая.
- Батч при фатальной ошибке → **фоллбек на построчный перевод чанка** (`translators.py:248-256`).
- Финальный фоллбек: если перевод не удался — возвращается **исходный английский текст** (файл не ломается).
- **Проактивного rate-limiting нет.** `RATE_LIMIT_SLEEP = 0.4` (`config.py:73`) объявлен, но **нигде не используется** (grep подтверждает). Запросы летят так быстро, как позволяет пул потоков.

## 2. Зависимость от ChatGPT

### 2.1 Жёсткие точки завязки на OpenAI

| Что | Где | Комментарий |
|-----|-----|-------------|
| Дефолтный base_url `https://api.openai.com` | `translators.py:138` | Читается через `getattr(config, "OPENAI_BASE_URL", ...)`, но **`OPENAI_BASE_URL` в `config.py` НЕ определён** → всегда дефолт. Env-переменной для него тоже нет. |
| Путь эндпоинта `/v1/chat/completions` | `translators.py:306`, `341` | Хардкод в двух местах. |
| Заголовок `Authorization: Bearer <key>` | `translators.py:330`, `368` | OpenAI-стиль. |
| Формат payload (`messages`, роли `system/user`) | `translators.py:318-325`, `356-363` | Chat Completions schema. |
| Формат ответа `choices[0].message.content` | `translators.py:337`, `376` | Chat Completions schema. |
| Ключ: `OPENAI_API_KEY` (env / `secrets.json` / `config`) | `translators.py:123`, `config.py:46,58-66` | Имя переменной жёстко «OPENAI». |
| Модель по умолчанию `gpt-4o-mini` | `config.py:44` | |
| GUI подписан «OpenAI API Key» | `gui_main.py:169-188` | Сохраняет ключ в `secrets.json`. |

**Хорошая новость для замены:** транспорт — это чистый OpenAI **Chat Completions** протокол поверх `urllib`. Любой OpenAI-совместимый эндпоинт (Ollama `/v1`, LM Studio, DeepSeek, Qwen/DashScope-compat, vLLM, together, groq) заработает **сменой одного base_url + ключа + имени модели**, если это значение прокинуть в конфиг/GUI.

### 2.2 ChatGPT-специфичные фичи

- **Не используются** function calling, tools, `json_mode`/`response_format`, `logprobs`, `seed`, assistants API.
- Из параметров — только `temperature: 0`. JSON-массив на выходе обеспечивается **текстовым промптом**, а не structured output — значит совместимо с любой моделью, но менее надёжно.
- OpenAI Python SDK (`openai==2.7.1` в `requirements.txt:10`) **в коде не импортируется вообще** (grep: ни одного `import openai`). Это **мёртвая зависимость** — код ходит голым `urllib`.

### 2.3 Объём запросов на модпак

Прямых счётчиков токенов в коде нет. Оценка по устройству пайплайна:

- Один batch = до `BATCH_SIZE = 100` уникальных строк (`config.py:9`; в конструкторе дефолт 50, но config переопределяет на 100 — `translators.py:133`).
- Дедуп по уникальным строкам + кэш (`translations_cache.json`) резко снижают повторные обращения; текущий кэш — **10 434 записи**.
- Средний модпак: типично **5 000–30 000** уникальных переводимых строк (lang-файлы модов + FTB Quests + Patchouli + tips). Это грубо **50–300 batch-запросов** + отдельные одиночные запросы на «сложные» строки (длинные, многострочные, с `§`/JSON).
- «Сложные» строки (FTB-квесты, лор, книги) идут **по одной** (`translators.py:222-228`) — они дают непропорционально много запросов.

## 3. Качество перевода

### 3.1 Консистентность терминов

- **Единственный механизм — кэш** (`src/utils/cache.py`): точное совпадение `исходная строка → перевод`. Гарантирует, что одинаковая английская строка всегда переводится одинаково в рамках прогонов.
- **Глоссария/термбазы нет.** Нет управляемого словаря («ingot → слиток» и т.п.). Разные, но родственные строки переводятся независимо → возможен разнобой терминов между модами.
- Кэш — **плоский словарь без версии/метаданных** (нет модели, даты, целевого языка внутри записи). Ключ кэша — сырой текст; при смене `TARGET_LANG` кэш **не различает языки** (см. риск R-3).

### 3.2 Контекст и форматирование Minecraft

- **§-коды:** `snbt_structured.py:37-45` (`_split_minecraft_formatting`) отрезает ведущие `§a§o…` и переводит только «ядро», затем приклеивает префикс назад. В самом переводчике `§`-строки помечаются «сложными» и уходят одиночками (`translators.py:226`).
- **Плейсхолдеры / namespaced ID:** двухуровневая защита — в промпте (не трогать) + пост-валидация `_extract_tokens` с откатом на оригинал (`translators.py:20-36, 176-179`).
- **Chat-JSON компоненты** (`{"text": "...", "with": [...]}`): `snbt_structured.py:167-208` рекурсивно парсит JSON, переводит только поля `text`, сохраняет структуру.
- **Контекст соседей отсутствует.** Каждая строка/батч переводится изолированно; модель не видит имя предмета рядом с его описанием, название мода, ключ строки. Ключи lang-файла (`item.modid.foo`) в промпт **не передаются** — теряется полезный контекст.
- **Путь как контекст** используется только для *фильтрации* (переводить/нет), не как подсказка модели (`snbt_structured.py:62-124`).

### 3.3 Пост-обработка / валидация

- Есть: сверка плейсхолдеров и namespaced ID (откат на оригинал при расхождении).
- Есть: пропуск уже-русских строк (`_looks_russian_only`, `translators.py:39-41`) и строк без латиницы.
- Есть: для batch — контроль длины/порядка массива.
- **Нет:** проверки на «модель вернула английский», обрезку кавычек-артефактов, контроль длины (переполнение UI-строк), нормализацию §-кодов на выходе, проверку сбалансированности скобок JSON после перевода.

## 4. Структура данных

### 4.1 Форматы входных файлов

| Формат | Детект | Процессор |
|--------|--------|-----------|
| `assets/*/lang/en_us.json` (моды, kubejs) | `mirrorer.py:19-20` | `lang_json.py` |
| `assets/*/lang/en_us.json` **внутри `.jar`** | `mirrorer.py:228-240`, `scanner.py:110-126` | `jar_lang.py` / `lang_json.translate_lang_obj` |
| Patchouli books `.../patchouli_books/**/en_us/**.json` | `mirrorer.py:23-29` | `generic_json.py` |
| Tips `.../tips/*.json` | `mirrorer.py:32-33` | `generic_json.py` |
| FTB Quests `.../ftbquests/**.snbt` | `mirrorer.py:36-37` | `snbt_structured.py` (осн.) + `ftb_snbt.py` (fallback regex) |
| KubeJS `.../kubejs/**.js` | `mirrorer.py:40-41` | `kubejs_js.py` (regex по вызовам `Text.of/tell/...`) |

`.properties`, `.toml`, `.lang` (старый плоский формат до 1.13) **не поддерживаются**.

### 4.2 Хранение исходников и переводов

- Исходники не хранятся — читаются из входной папки/`.jar` на лету.
- Переводы пишутся в выходное дерево (`out_root`), зеркалируя структуру (`en_us` → `ru_ru`).
- **Кэш** — единственное персистентное хранилище переводов: `translations_cache.json` (плоский JSON, `cache.py`). БД нет.
- Кэш пишется часто: `save()` после каждого батча/строки (`translators.py:183, 269`) — на больших прогонах это **много перезаписей файла 1.2 МБ**.

### 4.3 Инкрементальный перевод

- **Уровень файла:** `_dst_exists` (`mirrorer.py:54-67`) — если целевой файл уже существует, файл **пропускается целиком** (`[SKIP][exists]`). Это грубый инкремент: если исходный `en_us.json` изменился (добавились строки), файл **не перепереведётся**, пока пользователь не удалит выход.
- **Уровень строки:** кэш даёт инкремент по строкам между прогонами (изменённые строки переводятся заново, неизменённые берутся из кэша).
- **Нет** хэшей/mtime исходников, нет отслеживания «что изменилось с прошлого раза» на уровне контента файла.

## 5. Производительность и стоимость

### 5.1 Узкие места

- **Параллелизм — по файлам, не по запросам.** `ThreadPoolExecutor(max_workers=MAX_WORKERS_FILES=6)` (`mirrorer.py:298`, `config.py:8`). Внутри одного файла батчи идут **последовательно** (`translators.py:244`).
- **JAR-моды обрабатываются последовательно** после пула (`mirrorer.py:318-325`) — часто это самый большой объём (lang-файлы всех модов).
- **Батчинг есть** (до 100 строк/запрос) — это главный ускоритель и главный экономитель токенов.
- **`urllib` синхронный**, нет keep-alive пула соединений — каждый запрос открывает новое TLS-соединение (SSL-контекст переиспользуется, но не HTTP-пул). httpx/requests с пулом были бы быстрее.
- Частое `cache.save()` (полная сериализация 1.2 МБ после каждого батча) — **I/O-узкое место** на длинных прогонах.
- «Сложные» строки одиночками (`translators.py:222-228`) — для квестовых паков это может быть тысячи одиночных запросов.

### 5.2 Оценка стоимости на модпак (текущая реализация, `gpt-4o-mini`)

Грубо, для модпака ~15 000 уникальных строк (после дедупа и с холодным кэшем):

- Вход: ~15k строк × (~30 ток. текста + amort. ~4 ток. системного промпта при батче по 100) ≈ **~0.5–0.8 млн вход. токенов**.
- Выход (русский длиннее): ≈ **~0.7–1.0 млн выход. токенов**.
- Цена `gpt-4o-mini` ≈ $0.15/1M вход, $0.60/1M выход → **≈ $0.5–0.8 за модпак** при холодном кэше.
- На `gpt-4o`/`gpt-4.1` это было бы в ~15–30× дороже (**$10–25**).
- С тёплым кэшем повторные прогоны почти бесплатны.

> Это порядок величины, а не точный счёт — реального токен-учёта в коде нет (см. quick win Q-4).

## 6. Готовность к замене провайдера

### 6.1 Насколько абстрагирован слой LLM

**Скорее хорошо, чем плохо.** Вся сетевая логика инкапсулирована в `Translator` (`translators.py`); процессоры зависят только от методов `translate` / `translate_many`. Это значит: чтобы сменить провайдера, менять надо **один класс**.

**Но абстракция неполная:**
1. `OPENAI_BASE_URL` читается (`translators.py:138`), но **не определён в config и не прокинут из env/GUI** → сменить эндпоинт без правки кода нельзя.
2. `self.provider` **сохраняется, но нигде не ветвит логику** (`translators.py:121`) — заявленные в комментарии режимы `"ollama"` / `"dry"` (`config.py:43`) **не реализованы**. `OLLAMA_BASE_URL` (`config.py:47`) не используется.
3. Ключ и заголовок жёстко в OpenAI-стиле; для провайдеров без ключа (локальный Ollama) пустой `Bearer` обычно ок, но не гарантированно.
4. URL/схема ответа продублированы в двух методах — при рефакторинге легко забыть одно место.

### 6.2 Что нужно, чтобы работало с любым OpenAI-совместимым API

Минимальный путь (эндпоинт уже совместим):
1. Определить `OPENAI_BASE_URL` (или нейтрально `LLM_BASE_URL`) в `config.py` + env + поле в GUI.
2. Прокинуть его в `Translator.base_url` (уже читается — нужно только чтобы значение существовало).
3. Разрешить произвольное имя модели и пустой ключ (для Ollama/LM Studio).
4. Готово: DeepSeek, Qwen (compat), Ollama `/v1`, LM Studio, vLLM, groq, together заработают.

Более чистый рефактор:
1. Ввести конфиг-объект провайдера: `{base_url, api_key, model, extra_headers, auth_style}`.
2. Реализовать заявленные режимы `provider ∈ {openai_compatible, ollama_native, dry}`; `dry` = эхо исходника (полезно для тестов пайплайна без затрат).
3. Вынести построение URL/payload/парсинг ответа в один приватный метод `_chat(messages)` — устранить дублирование `_request_single`/`_request_batch`.
4. Убрать неиспользуемый `openai` SDK из `requirements.txt` (либо, наоборот, перейти на него — он даёт retry/pooling/timeout из коробки и работает с любым `base_url`).

## 10. Итоговый вывод

- **Замена ChatGPT реалистична и относительно дёшева.** Транспорт — чистый OpenAI **Chat Completions** поверх `urllib`, без ChatGPT-специфичных фич (нет function calling, json_mode, tools). Любой OpenAI-совместимый провайдер, включая **локальные Ollama/LM Studio и DeepSeek/Qwen**, подключается по сути **сменой `base_url` + модели + ключа**.
- **Главный блокер — не архитектура, а недоделанная конфигурация:** `OPENAI_BASE_URL` читается, но не прокинут; режимы `ollama/dry` заявлены, но не реализованы (**Q-1, Q-6** решают быстро).
- **Обязательно до миграции:** починить R-1 (битый вызов `process_jar`), ротировать ключ из `secrets.json` (R-2), сделать кэш многоязычным (R-3).
- **Для качества уровня «человеческая локализация»** нужны глоссарий (A-4) и контекст в промпте (A-5) — сейчас их нет, и это главный потолок качества независимо от выбранной модели.
